Branches

Git Repositorys benutzen Branches (dt. Äste), um Veränderungen wenn nötig zu isolieren. Es ist übliche Praxis in gemeinsamen Projekten, Änderungen in einem eigenenBranch (Ast) zu erstellen. Auf diese Weise können Änderungen getätigt und entwickelt werden, während der Hauptast (üblicherweiße 'master' genannt) stabil bleibt. Wenn die Arbeit im Nebenast dann stabil und beendet ist, wird der Nebenast wieder in den Hauptast eingefügt und verschmolzen (engl. merge).

Das folgende Diagramm zeigt, wie Du von Deinem Hauptast('master'-Branch) abzweigen kannst, im Nebenast arbeiten und dann diese Änderungen wieder in den Hauptast integrieren kannst. Du kannst wenn nötig sogar von einem Nebenast weitere Äste abzweigen, Basis muss hierfür nicht zwangsläufig der Hauptast sein.

Eine gute Visialisierung für die Funktion von Branches in einem Projekt ist im GitHub-Guide zu finden (engl.):guides.github.com/overviews/flow

GitHub Webseiten

GitHub hostet automatisch statische Webseiten-Dateien, die in Branches mit dem Namen 'gh-pages' abgelegt sind. Dieser kostenlose Service ist als GitHub Pages (dt. GitHub Seiten) bekannt. Da das in der letzen Aufgabe abgespaltete Projekt 'patchwork' eine Webseite erstellt, ist der Hauptast in diesem Projekt auch als 'gh-pages' benannt, anstatt 'master'. Alle Repositorys, die einen 'gh-pages'-Branch mit Webseiten-Dateien besitzen, sind live online unter dem folgenden URL-Muster:

http://GitHubNutzername.github.io/RepositoryName

Erzeuge einen Branch

Wenn Du einen neuen Branch erzeugst, kopiert Git automatisch alles aus dem aktuellen Branch und speichert es in dem neu erstellten Branch ab.

Gib im Terminal - noch immer in Deinem lokalen 'patchwork'-Repository - den Befehl git status ein, um zu sehen in welchem Branch Git aktuell arbeitet. Git sollte Dir den Branch 'gh-pages' zurückgeben.

Erzeuge nun einen neuen Branch mit der Bezeichnung "add-<NutzerName>", wobei 'NutzerName' deinen GitHub-Nutzernamen darstellen soll (Bsp. "add-jlord"). Achtung:Branches sind 'case-sensitive', das heißt Groß- und Kleinschreibung müssen unterschieden werden! Benenne Deinen jetzigen Branch im Repository exakt nach Deinem GitHub Nutzernamen!

git branch <BRANCHNAME>

Jetzt existert in Deinem Repository ein neuer Branch, der inhaltlich dem Branch 'gh-pages' entspricht.

Um in diesem Branch zu arbeiten, musst Du den Branch auswählen. In Git wird hierfür der Befehl checkout (dt. austesten) verwendet. Wähle Deinen neuen Branch aus:

git checkout <BRANCHNAME>

Nächster Schritt: Erstelle eine neue Datei

Im Text-Editor:

  • Erzeuge eine Datei mit dem Namen "add-<NutzerName>.txt", worin 'NutzerName' wieder Deinen GitHub-Nutzernamen bezeichnet. (Im bereits genannten Bsp.: "add-jlord.txt")
  • Als Inhalt der Textdatei schreibe einzig und alleine Deinen GitHub-Nutzernamen in die Datei. (Im Bsp. 'jlord').
  • Speichere diese Datei im Ordner 'contributors' (dt. Mitwirkende) im Patchwork-Verzeichnis: Patchwork/contributors/add-NutzerName.txt
  • Diese Änderungen kannst Du nun im Repository sichern.

Änderungen im Repository sichern

Gehe durch die folgenden Befehle, um Deine Änderungen am Projekt zu speichern:

git status

git add <contributors/DATEINAME>

git commit -m "Commit Nachricht"

Schiebe jetzt Deine Aktualisierung in Deinen Fork 'origin' bei GitHub:

git push origin <BRANCHNAME>

Permission denied...error: 403

'Permission denied' (dt. Erlaubnis verweigert) bezeichnet den Fehler, wenn Du versuchst auf ein Repository zu schieben zu dem Du keine Änderungs-Berechtigung besitzt. In unserem Fall versuchst Du vermutlich auf das Original-Repository 'jlord/patchwork' zu schieben. Stelle sicher, dass Du zum Remote 'origin' schieben möchtest, und dass dieser Remote auf die Adresse Deines Forks bei GitHub zeigt. Um Deine Remotes anzuschauen und zu überprüfen wohin diese zeigen nutze den Befehl git remote -v. Du solltest hier den Remote 'upstream' sehen, der auf 'jlord/patchwork' zeigt, sowie den Remote 'origin', der auf 'DeinNutzerName/patchwork' zeigt.

Um den Fehler eines fehlgeleiteten Remote zu beheben und den Zeiger des Remotes neu zu setzen nutze folgenden Befehl: git remote set-url origin .

Authentication failed...error: 401

'Authentication failed' (dt. Authetifizierung fehlgeschlagen) bezeichnet den Fehler, dass Deine Identität nicht verifiziert werden konnte. Falls du die zwei-Faktor-Authetifizierung aktiviert hast (2FA - Two Factor Authentication), muss ein Access-Token als Passwort verwendet werden. Generiert werden kann dieser durch verfolgen der Anweisungen in der GitHub Hilfe. Dieser Access-Token fungiert dann als Dein Passwort - bewahren ihn also gut auf! Du kannst den Access-Token auch auf Deinem Computer speichern, damit Du ihn nicht immer wieder eingeben musst.

File NOT in contributors folder

'File NOT in contributors folder' (dt. Datei nicht im Ordner 'contributors') - Die erstellte Datei sollte im 'contributors'-Verzeichnis in Deinem Patchwork-Repository gespeichert sein. Solltest Du sie an einem anderen Ort abgespeichert haben, benutze am besten die Suche und schiebe die Datei in das 'contributors'-Verzeichnis. Teste anschließend erneut den git status füge die Datei in Git den Änderungen hinzu (oder füge alle Änderungen hinzu: git add -A) und übergib die Änderungen an die Historie:

git add -A

git commit -m "Datei in das contributors Verzeichnis verschoben"

Branch name expected: _____

'Branch name expected: _____' (dt. Branch-Name erwartet:) Der Branch-Name sollte exakt Deinem Nutzernamen entsprechen. Um den Branch-Namen zu ändern nutze den Befehl:

git branch -m <NEUERBRANCHNAME>

Wenn Du diese Aktualisierungen gemacht hast, kannst Du Deinen Erfolg erneut prüfen lassen!

  • Du kannst einen Branch mit einem einzigen Kommando erzeugen und auswählen
  • git checkout -b <BRANCHNAME>
  • Erzeugen einen neuen Branch
  • git branch <BRANCHNAME>
  • Wähle einen Branch aus
  • git checkout <BRANCHNAME>
  • Liste die verfügbaren Branches auf
  • git branch
  • Benenne den akutell ausgewählten Branch um
  • git branch -m <NEWBRANCHNAME>
  • Schaue, auf welchem Branch Du aktuell arbeitest
  • git status
