Verbinde Dein lokales Repository mit einem entfernten Repository und schiebe Änderungen dorthin.

Remotes

Wenn Du etwas auf GitHub speicherst, liegt diese Kopie auf einem der Server von GitHub. Das macht es zu einem Remote-(dt. Fern-)Repository, da es nicht auf Deinem Computer gespeichert ist, sondern auf einem entfernten Server - "ferngesteuert" und irgendwo in der Welt. Durch schieben (engl. push) Deiner lokalen(auf Deinem Computer) Änderungen auf den Server hältst Du die Dateien auf dem Server aktuell.

Andere können dann immer die aktuelle Version Deines Projektes bekommen, indem sie Deine Änderungen vom Remote ziehen (engl. pull) und auf ihrem Computer speichern. Somit kann jeder an einem Projekt arbeiten, ohne direkten Zugang zu dem Computer zu haben, auf dem Deine lokale Kopie gespeichert ist.

Erzeuge ein Remote Repository

Du willst nun die lokale Version Deines Repositorys mit einem Repository synchronisieren, das auf GitHub.com gespeichert ist. Dazu erstelle zuerst ein neues Remote Repository auf GitHub.com.

  • Öffne github.com, melde Dich an, klicke auf das '+' in der rechten oberen Ecke und klicke dann auf 'New repository'.
  • Gib dem neuen Repository den selben Namen, wie Deinem lokalen Repository ('hello-world') und füge eine kurze Beschreibung hinzu.
  • Speichere das Repository als 'public' (dt. öffentlich). Das Repository wird somit auf Deinem öffentlichen Profil zu sehen sein.
  • Initialisiere das Repository nicht mit einer README-Datei, Du hast bereits lokal eine Datei mit dem Namen 'readme.txt' gespeichert. Es handelt sich hier um eine Hilfsfunktion von GitHub, falls Du die Datei nicht selbst anlegst.
  • Lasse die Optionen '.gitignore' (dt. gitignorieren) und 'license'(dt. Lizenz) eingestellt als 'none'(dt. keine). Wir werden diese Dateien in den Aufgaben nicht nutzen.
  • Klicke auf 'Create repository' (dt. Repository erzeugen)!

README, .gitignore und Lizenzen

'README' (dt. Liesmich), '.gitignore' (dt. .gitignorieren) und Lizenzdateien sind übliche Dateien in einem Open-Source Projekt. Um den Projektstart etwas einfacher zu machen stellt GitHub daher die Möglichkeit bereit, diese Dateien für Dich zu erzeugen.

Eine README-Datei erklärt das Projekt, wie es zu nutzen ist und oft auch wie man zum Projekt beitragen kann. (Letztere Infos werden manchmal auch in einer extra Datei 'CONTRIBUTING.md' (dt. beitragen) bereitgestellt.)

Die Datei .gitignore stellt eine Liste von Dateien dar, die Git nicht nachverfolgen soll, was z.B. Dateien mit Passwörtern sehr nützlich ist.

Eine Lizenz-Datei bestimmt die Lizenz unter der Dein Projekt verbreitet werden darf. Andere können daraus erfahren, unter welchen Bedingungen sie das Projekt nutzen und weiterverbreiten dürfen. Informationen zu den verschiedenen Lizenztypen findest Du z.B. hier (engl.):choosealicense.com.

Verbinde das Lokale mit dem Remote-Repository

Du hast nun ein leeres Repository auf GitHub.com erstellt. Im oberen Teil der Seite solltest Du nun den Abschnitt 'Quick Setup' (dt. Schnelleinrichtung) sehen. Stelle sicher, dass 'HTTPS' ausgewählt ist und kopiere dann die angezeigte Adresse. Dies ist die Adresse Deines Repository auf dem GitHub-Server.

Zurück im Terminal und in Deinem 'hello-world' Verzeichnis (das Du schon als Git Repository initialisiert hast) willst Du Git nun die Adresse des zu verknüpfenden Remote-Repository auf dem GitHub Server mitteilen. Da es möglich ist mehrere Remotes zu verknüpfen, müssen diese eindeutig benannt werden. Das primäre Remote-Repository wird dabei meist mit origin (dt. Ursprung) bezeichnet.

Um ein Remote mit dem Namen 'origin' zu Deinem Repository hinzuzufügen:

git remote add origin <URLVONGITHUB>

Dein lokales Repository weiß nun, wo Dein Remote-Repository mit dem Namen 'origin' auf dem GitHub-Server liegt. Du kannst es Dir vorstellen, wie einen Namen und eine Adresse zur Kurzwahl hinzuzufügen. Nun kannst Du dorthin etwas senden, wenn du möchtest.

Windows Benutzer:

Wenn GitHub Desktop auf Deinem Computer installiert ist, wird in Deinem lokalen Repository automatisch ein Remote mit dem Namen 'origin' erzeugt. In diesem Fall musst Du Git nur noch die zugehörige URL mitteilen, die mit 'origin' verknüpft werden soll. Statt dem Befehl 'add' kannst Du dieses Kommando verwenden:

git remote set-url origin <URLFROMGITHUB>

Schiebe Deine Arbeit zum Remote

Als nächstes möchtest Du alles, was du lokal gemacht hast zu Deinem Remote-Repository bei GitHub schieben (engl. push). Dies ist eine Funktion, die Du sehr oft nutzen wirst, damit Deine Remote-Version immer aktuell ist und dem Status Deiner lokalen Version entspricht.

Git besitzt ein Verzweigungs-System, mit dem es möglich ist an verschiedenen Teilen des Projektes parallel zu arbeiten. Wir werden später mehr darüber lernen, aber standardmäßig wird der erste Branch (dt. Ast) mit dem Namen 'master' bezeichnet. Wenn Du aus einem Projekt heraus Dateien auf den Server schiebst (und später auch von dort ziehst), nennst Du Git den Branch-Namen zu dem Du schieben möchtest, sowie den zugehörigen Remote.

In diesem Fall senden wir unseren Branch 'master' zum dem Remote auf GitHub mit dem Namen 'origin'.

git push origin master

Gehe nun zurück zu Deinem Remote-Repository auf GitHub.com und aktualisiere die Seite. Wow! Online sind nun die selben Daten verfügbar, wie in deinem lokalen Verzeichnis. Gratulation zu Deinem ersten öffentlichen Repository!

    • git remote set-url <REMOTENAME> <URL>

    • git pull <REMOTENAME> <BRANCHNAME>

    • git remote -v
  • Schiebe Änderungen auf den Server
    • git push <REMOTENAME> <BRANCH>
