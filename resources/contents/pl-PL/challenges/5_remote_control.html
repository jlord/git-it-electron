Połączenie lokalnego repozytorium ze zdalnym i wypchnięcie zmian.

Repozytoria zdalne

Kiedy umieszczasz coś na GitHub ta kopia żyje na jednym z serwerów do niego należących. To czyni je repozytorium zdalnym gdyż nie znajduje się ono na Twoim komputerze, ale na serwerze, "zdalnie" i gdzieś indziej. Wypychając swoje lokalne (znajdujące się na Twoim komputerze) zmiany, dokonujesz jego aktualizacji.

Inni mogą zawsze pobrać na swój komputer najaktualniejsze zmiany z twojego projektu wprost z repozytorium zdalnego. To właśnie tak wszyscy razem pracują nad jednym projektem bez konieczności uzyskiwania dostępu do Twojego komputera, gdzie przechowywana jest kopia lokalna.

Tworzenie repozytorium zdalnego

Chcesz zsynchronizować zmiany zawarte w lokalnej wersji repozytorium z tymi przechowywanymi na GitHub.com. Najpierw utwórz nowe repozytorium zdalne na GitHub.com.

  • Przejdź do github.com, zaloguj się i kliknij ikonę '+' w prawym górnym rogu, a następnie wybierz opcję 'New repository'.
  • Nadaj jemu nazwę odpowiadającą nazwie Twojego lokalnego repozytorium, 'hello-world', oraz krótki opis.
  • Wybierz repozytorium publiczne. Oznacza to, że będzie wyświetlane w Twoim profilu oraz widoczne dla każdej osoby je odwiedzającej.
  • Nie inicjuj repozytorium plikiem README ponieważ już taki mamy lokalnie - 'readme.txt'. Jest to opcja pomocnicza GitHuba, przydatna jeśli jeszcze nie masz takiego pliku.
  • Pozostaw '.gitignore' i 'license' ustawione na 'none'. Nie będziemy korzystać z nich w ramach tego poradnika.
  • Kliknij przycisk by zatwierdzić utworzenie repozytorium!

Readme, .gitignore i licencje

Są to pliki popularne w projektach open source, więc by ułatwić utworzenie nowego projektu, GitHub oferuje utworzenie ich w Twoim imieniu.

Readme ('czytaj to') wyjaśnia czym jest projekt, jak z niego korzystać, a czasami także jak włączyć się do pracy na rzecz jego rozwoju (choć często jest to wyjaśniane w dodatkowym pliku 'CONTRIBUTING.md', tworzonym specjalnie na taką okazję).

.gitignore zawiera listę plików których Git nie powinien śledzić, takie jakie pliki zawierające hasła czy dane dostępowe do systemów produkcyjnych.

license zawiera rodzaj licencji jaka udzielana jest na projekt. Pozwala to innym na zapoznanie się z zasadami jego użytkowania. Informacje odnośnie rodzajów licencji można znaleźć tutaj: choosealicense.com.

Podłączanie repozytorium lokalnego do repozytorium zdalnego

Teraz posiadasz już utworzone puste repozytorium zdalne na on GitHub.com. Na górze znajdziesz 'Quick Setup', upewnij się, że wybrano opcję 'HTTPS' i skopiuj adres - to jest adres Twojego repozytorium na serwerach GitHuba.

W swoim terminalu, w folderze 'hello-world' gdzie zainicjowałeś repozytorium w poprzednim wyzwaniu, chcesz powiedzieć Gitowi by zapamiętał adres zdalnej wersji tego repozytorium. Ponieważ możesz mieć wiele repozytoriów zdalnych, każde z nich wymaga osobnej nazwy. Główne repozytorium zdalne jest często nazywane origin.

W celu dodania repozytorium zdalnego 'origin' do wersji lokalnej:

git remote add origin <URLDOGITHUBA>

Twoje repozytorium lokalne wie już gdzie szukać repozytorium zdalnego, nazwanego 'origin'. Pomyśl o tym jak o dodawaniu nazwy i numeru telefonu do szybkiego wybierania, jeśli potrzebujesz się z kimś szybko skontaktować, wystarczy po prostu wybrać odpowiednią opcję przypisaną do klawisza.

Użytkownicy Windowsa:

Jeśli posiadasz na swoim komputerze aplikację GitHub Desktop, zdalne repozytorium nazwane 'origin' jest automatycznie dodawane do lokalnego repozytorium. W tej sytuacji jedyne co musisz zrobić to poinformować jaki adres URL z nim powiązać. Użyj poniższego polecenia zamiast podanego powyżej 'add':

git remote set-url origin <URLDOGITHUBA>

Wypychanie zmian do repozytorium zdalnego

Następnie chcesz wypchnąć (wysłać) wszystkie zmiany lokalne do repozytorium zdalnego na GitHub. To jest coś, co będziesz robić często by wersja zdalna pozostała zawsze zaktualizowana o lokalne zmiany.

Git posiada system gałęzi, co umożliwia skoncentrowanie prac nad różnymi aspektami projektu w różnych chwilach. Więcej dowiemy się o tym później, ale domyślnie pierwsza gałąź nazywana jest 'master'. Kiedy wypychasz (i później pobierasz) zmiany, musisz powiedzieć gitowi z której gałęzi ma skorzystać używając jej nazwy oraz nazwy repozytorium zdalnego na którym ona istnieje.

W tym przypadku, wyślemy naszą gałąź nazwaną 'master' na nasze repozytorium GitHub nazwane 'origin'.

git push origin master

Teraz przejdź na stronę swojego repozytorium zdalnego na GitHub.com i odśwież stronę. Wow! Wszystko jest teraz zsynchronizowane. Gratulacje utworzenia pierwszego repozytorium zdalnego!
