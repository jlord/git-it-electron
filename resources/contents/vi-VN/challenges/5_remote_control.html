Kết nối repos cục bộ của bạn với một remote (điều khiển từ xa) và đưa lên Git những thay đổi bạn đã làm.

Remotes

Khi bạn đưa thứ gì đó lên GitHub, máy chủ GitHub sẽ sao lưu ngay một bản. Đó là lí do ta có remote của repository bởi nó không nằm trên máy tính của bạn mà nằm trên một server, "remote" (điều khiển từ xa) và nơi nào đó. Với việc push các repos local (trên máy tính) và thay đổi nó, bạn giữ cho việc cật nhật nó được diễn ra mọi lúc có thay đổi.

Người khác cũng có thể lấy được những thay đổi mới nhất từ project của bạn bằng việc pull về các thay đổi của bạn thông qua remote (và trên máy của họ). Đây là cách mà mọi người làm việc cùng một project với nhau mà không cần phải truy cập vào máy tính của bạn - nơi chứa repos.

Tạo một Repos Remote

Bạn muốn đồng bộ local version với một repos đã lưu trên GitHub.com. Đầu tiên, tạo ra mộtremote repository mới trên GitHub.com.

  • Đi đến github.com, đăng nhập, và click biểu tượng '+' ở phía trên bên phải rồi chọn 'New Repository'.
  • Đặt cho repos đó một cái tên trùng với repos cục bộ của bạn, 'hello-world', và một vài mô tả ngắn trong description.
  • Đặt nó ở trạng thái công khai. Điều này làm cho repos đó sẽ được hiển thị trên profile của bạn công khai.
  • Không cần tạo file README vì ta đã có sẵn file trong repos cục bộ có tên là 'readme.txt'. Đây chỉ là một lựa chọn mà GitHub hỗ trợ nếu bạn chưa tạo file README.
  • Bỏ qua '.gitignore' và 'license' đặt là 'none'. Ta chưa cần dùng đến trong challenge lần này.
  • Click tạo repos - create repository!

README, gitignores and Licenses

Có những file thường thấy trong các dự án open source, nên để dễ dàng cho việc bắt đầu một project mới, GitHub cung cấp một lựa chọn tạo ra các file đó cho bạn.

Một file readme giải thích về project: project đó là gì, sử dụng nó như thế nào, và thường là làm sao để cộng tác(contribute) (đôi khi mục này được tách ra một file, có tên là 'CONTRIBUTING.md', để chi tiết hơn).

Một file.gitignore là một danh sách của các file mà Git không phải để ý, theo dõi thay đổi files with passwords!

Một giấy phép - license file là một dạng giấy phép bạn đặt cho project của mình. Nó chứa thông tin cho người khác biết cách sử dụng nó. Thông tin của dạng này bạn có thể tìm ở đây: choosealicense.com.

Kết nối repos cục bộ đến Remote

Giờ bạn đã có một repository trắng trên GitHub.com. Ở phía trên bạn sẽ thấy 'Quick Setup', đảm bảo rằng nút 'HTTPS' đã được chọn và sao chép địa chỉ đó, đó chính là địa chỉ repos của bạn trên máy chủ GitHub.

Quay trở lại với terminal và repos 'hello-world' cục bộ mà bạn đã tạo Git repos trong challenge trước đó, bạn cần cho Git biết và nhớ địa chỉ của remote version trên máy chủ GitHub. Bạn có thể có nhiều remote và mỗi remote yêu cầu một tên. Ở dạng nguyên thủy, tên thường được đặt là origin.

Để thêm remote có tên 'origin' vào repos:

git remote add origin <URLFROMGITHUB>

Repos trên local của bạn giờ đã biết remote repos có tên 'origin' trên máy chủ GitHub nằm ở đâure. Có thể liên tưởng với việc bạn thêm tên và địa chỉ trong mục quay số nhanh khi bạn cần gửi đến nó, bạn chỉ việc gửi theo số quay số nhanh đã lưu.

Đối với người dùng Windows:

Nếu bạn có GitHub Desktop trên máy tính, một remote có tên 'origin' sẽ tự động được tạo trong repos cục bộ. Trong trường hợp này, bạn chỉ cần cho nó biết URL liên kết với 'origin'. Dùng lệnh này thay cho 'add':

git remote set-url origin <URLFROMGITHUB>

Đẩy lên Remote

Tiếp đó, bạn muốn push (gửi) mọi thứ bạn đã hoàn thành dưới local đến remote repos của bạn trên GitHub. Đây là việc bạn sẽ làm thường xuyên vì vậy remote version cập nhật và so khớp với trạng thái của local version.

Git có hệ thống rẽ nhánh nên bạn có thể làm việc với các phần khác nhau của project ở những thời điểm khác nhau. Ta sẽ học sâu về nó sau, nhưng mặc định, nhánh đầu tiên có tên là 'master'. Khi bạn đẩy (và sau đó là kéo về) từ một project, bạn sẽ cho Git biết branch name - tên nhánh mà bạn muốn và tên của remote mà bạn cần.

Trong trường hợp này, ta sẽ gửi đến branch có tên 'master' nằm trên remote trên GitHub tên là 'origin'.

git push origin master

Và giờ ta đi đến remote repos của bạn trên GitHub.com và chạy lại trang. Wow! Tất cả các file đã được cập nhật giống với local repos!. Vậy là bạn đã hoàn thành với repos đầu tiên!
